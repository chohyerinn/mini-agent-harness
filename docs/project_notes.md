# Project Notes

## Why I built this

처음에는 코딩 에이전트가 한 번 문제를 풀면 그 결과를 그대로 믿어도 되는 줄 알았다. 그런데 같은 프롬프트와 같은 테스트에서도 실행할 때마다 결과가 달라졌다. 어떤 실행은 통과하고, 어떤 실행은 edge case를 놓쳤다.

그래서 이 프로젝트는 “에이전트가 풀었는가?”보다 “여러 번 돌렸을 때 얼마나 안정적으로 푸는가?”를 보려고 만들었다. 단일 실행 결과가 아니라 solve rate, pass@k, 실패 유형, 토큰 비용, 실행 시간을 같이 남기는 작은 평가 하니스가 목표였다.

## What was difficult

가장 어려웠던 부분은 평가 결과를 믿을 수 있게 만드는 것이었다. 에이전트에게 테스트 파일을 처음부터 보여주면, 실제 버그를 고치지 않고 테스트에 맞춰서만 코드를 바꿀 수 있다. 또 `conftest.py`, `pytest.py`, `pyproject.toml` 같은 파일을 수정하면 pytest 동작 자체가 바뀔 수 있었다.

두 번째로 어려웠던 부분은 멀티에이전트 결과를 해석하는 일이었다. Planner-Coder-Reviewer 구조가 단일 에이전트보다 좋아 보일 때도 있었지만, 쉬운 문제에서는 오히려 reviewer가 과수정을 유발하거나 edge case를 놓쳤다. 단순 평균만 보면 좋아진 것처럼 보이는데, 실제로 비용과 통계적 유의성을 같이 보면 결론이 달라졌다.

## Issues I ran into

### 1. 한 번 성공한 결과를 믿기 어려웠다

처음에는 각 과제를 한 번만 실행했다. 하지만 LLM 출력은 비결정적이라 한 번 성공하거나 실패한 결과만으로는 에이전트의 실력을 말하기 어려웠다. 그래서 같은 과제를 여러 번 실행하고, solve rate와 pass@k를 기록하도록 바꿨다.

### 2. 테스트 변조 가능성이 있었다

에이전트가 작업 폴더 안에서 `conftest.py`나 `pytest.py`를 만들면 채점 환경을 바꿀 수 있었다. 이 경우 테스트가 통과해도 실제 버그를 고친 것인지 알 수 없다. 그래서 정답 테스트는 에이전트 실행 중에는 숨기고, 채점 직전에만 복사했다. 또한 테스트 해시를 비교하고, pytest 설정 파일이나 hook 파일이 새로 생기거나 바뀌면 tampering으로 처리했다.

### 3. 실제 OSS provenance를 확인하는 과정에서 실수가 있었다

OSS 기반 과제를 추가할 때 처음에는 PR과 관련 없는 commit SHA가 들어간 적이 있었다. 겉보기에는 실제 SHA처럼 보였지만, 클릭해보면 해당 버그 수정과 무관한 커밋이었다. 이 문제를 겪고 나서 `task.yaml`의 `source_ref`와 `source_commit`은 실제 PR/commit 페이지와 코드 변경 내용을 대조해서 넣는 방식으로 바꿨다.

### 4. 멀티에이전트가 항상 좋아지지 않았다

15개 과제를 10회씩 실행했을 때 단일 에이전트는 `75/150`, 멀티에이전트는 `83/150`을 해결했다. 숫자만 보면 멀티가 좋아 보이지만, paired bootstrap CI가 0을 포함했고 McNemar p-value도 `0.3123`이었다. 이 실험에서는 개선 방향은 보였지만 통계적으로 확정된 개선이라고 말하기 어려웠다.

### 5. 비용이 예상보다 크게 늘었다

멀티에이전트는 Planner, Coder, Reviewer를 나눠 호출하기 때문에 토큰 사용량이 크게 늘었다. 실제 실험에서는 solved run 하나당 토큰이 `1290.01 → 3884.6`으로 늘었고, 추가 성공 1건당 약 `28,209` tokens가 더 들었다. 그래서 “조금 더 많이 풀었다”만으로는 구조가 더 낫다고 말할 수 없었다.

### 6. CLOVA API rate limit과 긴 대기 시간이 있었다

멀티에이전트 실험은 API 호출 수가 많아서 429 rate limit을 만났다. 긴 실행 중에는 한 요청이 오래 대기하다가 중간에 끊긴 적도 있었다. 이 문제 때문에 retry/backoff, 실행 간 sleep 옵션, timeout 환경변수를 넣었다.

## How I fixed them

- `run_task`가 매 실행마다 독립 workspace를 만들도록 했다.
- agent 실행 중에는 `tests/`를 숨기고, 채점 직전에만 복사했다.
- pytest 실행 전후로 테스트 해시를 비교했다.
- `conftest.py`, `pytest.py`, `pyproject.toml`, `sitecustomize.py` 같은 변조 가능 파일을 감지했다.
- 환경변수 whitelist를 사용해서 테스트 프로세스에 API 키가 넘어가지 않게 했다.
- 반복 실행 결과를 task별로 모으고 pass@k, 평균, 표준편차를 계산했다.
- A/B 비교에는 bootstrap CI와 McNemar test를 넣어 “확정 개선”과 “개선 후보”를 나눴다.
- Docker sandbox 모드를 추가해서 원하면 pytest를 네트워크 없는 컨테이너에서 실행할 수 있게 했다.
- CLOVA 호출에는 retry/backoff와 sleep 옵션을 넣었다.

## What I learned

이 프로젝트를 하면서 “AI가 답을 냈다”보다 “그 답을 어떻게 믿을 것인가”가 더 어렵다는 걸 느꼈다. 특히 에이전트 평가는 정답률 하나로 끝나지 않는다. 반복 실행, 실패 유형, 비용, 실행 시간, 평가 환경 변조 가능성을 같이 봐야 한다.

또 멀티에이전트 구조가 항상 정답은 아니라는 것도 배웠다. 역할을 나누면 좋아지는 과제도 있지만, 쉬운 문제에서는 불필요한 수정이나 reviewer miss가 생길 수 있었다. 그래서 평균 성능보다 회귀와 비용을 같이 보는 게 중요했다.

마지막으로, provenance를 대충 적으면 안 된다는 것도 배웠다. “실제 OSS PR 기반”이라고 쓰려면 PR 링크와 commit SHA가 정말 그 버그를 고친 커밋인지 확인해야 한다.

## What I would improve next

- OSS 기반 과제를 더 늘리고 난이도별로 나눠보고 싶다.
- Docker sandbox를 더 오래 실행해보고, 컨테이너 이미지와 리소스 제한을 더 명확히 고정하고 싶다.
- CLOVA 외 다른 모델도 같은 과제군에서 비교해보고 싶다.
- 결과 추이를 볼 수 있는 작은 dashboard를 붙이면 장기 실험을 보기 쉬울 것 같다.
